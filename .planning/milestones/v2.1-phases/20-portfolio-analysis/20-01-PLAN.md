**Comment line:** `<!-- Sub-skill loaded by router SKILL.md. All CLAUDE_SKILL_DIR paths resolve to skills/insurance/ -->`

**`<objective>`:** Deliver a portfolio-level insurance analysis by spawning the portfolio agent. The sub-skill: (1) checks for existing policies in profile, (2) runs interactive policy entry if none exist, (3) emits disclaimer, (4) spawns portfolio agent via Task, (5) renders the analysis as tier-classified conversational output.

**`<process>`** with these phases:

**Phase 0: Policy Data Check**
- Read `.finyx/profile.json` (already loaded by router)
- Check `insurance.policies[]`
- If array is empty or key is absent, use AskUserQuestion with multiSelect:
  - Preamble: "You have no insurance policies recorded. Which insurance types do you currently have? Select all that apply, then we'll enter the details for each."
  - Options (exactly these 12):
    - Health insurance (GKV or PKV)
    - Personal liability (Haftpflichtversicherung)
    - Household contents (Hausratversicherung)
    - Car insurance (Kfz-Versicherung)
    - Legal protection (Rechtsschutzversicherung)
    - Dental supplement (Zahnzusatzversicherung)
    - Term life (Risikolebensversicherung)
    - Travel insurance (Reiseversicherung)
    - Bicycle insurance (Fahrradversicherung)
    - Roadside assistance (Kfz-Schutzbriefversicherung)
    - Rental deposit insurance (Mietkautionsversicherung)
    - None -- I want to see what I'm missing (gap analysis only)
  - If "None" selected: skip to Phase 1 with empty policies (agent will produce gap-only analysis)
  - For each selected type: use AskUserQuestion to collect provider name and monthly premium
  - Map type selections to slugs: Health->health, Personal liability->haftpflicht, Household contents->hausrat, Car->kfz, Legal protection->rechtsschutz, Dental->zahnzusatz, Term life->risikoleben, Travel->reise, Bicycle->fahrrad, Roadside->kfz-schutzbrief, Rental deposit->mietkaution
  - Build policy objects with ALL required schema fields: id (generate as `{type}-{provider_slug}-{year}`), type, provider, premium_monthly, premium_annual (premium_monthly * 12), coverage_amount (null), start_date (null), renewal_date (null), kuendigungsfrist_months (3), sonderkundigungsrecht (false), doc_path (null), coverage_components ([]), notes (null), last_updated (current ISO date)
  - Write updated profile.json with the new policies[] entries via Write tool
- If policies[] already has entries: proceed to Phase 1

**Phase 1: Disclaimer**
- Emit the header banner:
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 FINYX ► INSURANCE: PORTFOLIO OVERVIEW
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```
- Emit full legal disclaimer from the loaded `disclaimer.md` (loaded by router's execution_context)

**Phase 2: Spawn Portfolio Agent**
- Check profile for living area field: look for `investor.livingAreaSqm`, `criteria.minSize`, or any sqm-related field. If found, set `property_size_sqm` to that value. If not found, set to null.
- Spawn `finyx-insurance-portfolio-agent` via Task tool with this prompt:

```
You are the finyx-insurance-portfolio-agent. Analyze the user's full insurance portfolio.

Profile data is at `.finyx/profile.json`. Reference docs for all 11 insurance types are already loaded in your execution_context.

[If property_size_sqm is known:]
property_size_sqm: {value}

[If property_size_sqm is null:]
property_size_sqm: null — Hausrat adequacy will be marked UNKNOWN; advise user to add living area to profile via /finyx:profile.

Complete all phases of your process and return your output wrapped in <portfolio_analysis> tags.
```

- Collect the `<portfolio_analysis>` output

**Phase 3: Render Output**
- Parse the `<portfolio_analysis>` XML block from the agent
- Render as conversational advisory text organized by tier classification (Mandatory > Essential > Recommended > Situational) per locked decision
- Present sections in this order:
  1. Portfolio overview with total monthly cost
  2. Coverage adequacy table with benchmark source attribution (e.g., "Hausrat: >=650 EUR/m^2 per GDV recommendation")
  3. Gap detection table with priority and family-context reason (using identity.family_status, identity.children from profile)
  4. Overlap warning banners — each overlap as a separate warning with affected policies named inline per locked decision
  5. Cost summary per policy with monthly and annual totals
- If agent did not return `<portfolio_analysis>`: emit error "Portfolio agent output not received. Re-run `/finyx:insurance portfolio` to retry."
- If property_size_sqm was null: include note "Hausrat adequacy requires living area (Wohnflaeche). Add it via /finyx:profile for a complete assessment."

**`<error_handling>`:**
- No profile found: "ERROR: No financial profile found. Run /finyx:profile first to complete your financial profile."
- Portfolio agent fails: "Portfolio agent output not received — finyx-insurance-portfolio-agent did not return the expected `<portfolio_analysis>` block. Re-run `/finyx:insurance portfolio` to retry."

**`<notes>`:**
- Write Targets: This sub-skill writes to `.finyx/profile.json` ONLY during the interactive policy entry flow (Phase 0) when no policies exist. All other output is conversational only.
- Disclaimer Placement: Disclaimer emitted in Phase 1 before any advisory content, matching the health.md disclaimer-first pattern.
- Agent Spawning: Single agent spawn (portfolio agent). Unlike health.md which spawns two agents in parallel, portfolio uses only one agent.
- Property Size: The sub-skill checks profile for a living area field to pass as `property_size_sqm`. If not found, Hausrat adequacy is marked UNKNOWN by the agent.
- Sub-skill Tool Permissions: Read, Task, AskUserQuestion, Write (Write only for empty-portfolio policy entry)
