**process section (5 phases):**

Phase 1: Read Profile
- Read `.finyx/profile.json`
- Extract `insurance.policies[]` array
- If array is empty or missing: output "No insurance policies recorded. Run /finyx:insurance to add policies or use /finyx:profile to enter them manually." and STOP.
- Extract `identity.family_status`, `identity.children` for gap prioritization

Phase 2: Coverage Adequacy Check
- For each policy in policies[]: load the matching reference doc by type
- Read Coverage Benchmarks from the reference doc
- For sum_based types: compare policy.coverage_amount against benchmark threshold
- For service_based types: mark as "COVERED" if policy exists
- Output status per type: OK / BELOW BENCHMARK / UNKNOWN (if coverage_amount is null for a sum_based type)

Phase 3: Gap Detection
- Build list of all 11 insurance types
- For each type NOT in policies[]: check reference doc for legal minimum and priority
- Classify gaps: MANDATORY (legally required -- only Kfz Haftpflicht if user owns a car), ESSENTIAL (strongly recommended given life situation), OPTIONAL (situational)
- Use family_status and children to adjust priority (e.g., Risikoleben is ESSENTIAL if children > 0, OPTIONAL if single)

Phase 4: Overlap Detection
- Known overlap pairs (hardcode these -- they are domain knowledge):
  - Hausrat Fahrrad-Zusatz vs standalone Fahrrad policy
  - Reiseversicherung vs Kfz-Schutzbrief (some travel components overlap)
  - Kfz-Schutzbrief vs ADAC membership (not in policies[], but note it)
  - Rechtsschutz Verkehrsrechtsschutz component vs standalone Verkehrsrechtsschutz
- For each pair: check if both exist in policies[]. If yes, check coverage_components for overlap.

Phase 5: Format Output
- Wrap in `<portfolio_analysis>` tags
- Sections: Overview (total monthly/annual cost), Coverage Adequacy table, Gaps table, Overlaps table, Cost Summary table
- Use the exact output format from 19-RESEARCH.md "Portfolio Agent Output Format" section
- Include disclaimer reference

**FILE 2: finyx-insurance-doc-reader-agent.md (ARCH-04)**

YAML frontmatter:
- name: finyx-insurance-doc-reader-agent
- description: Extracts structured policy data from insurance PDF documents. Spawned by /finyx:insurance sub-skills.
- tools: Read, Grep, Glob
- color: yellow

**role section:**
- Identity: "You are a Finyx insurance document reader."
- Core job: "Given a doc_path (PDF file path) and insurance_type, load the matching reference doc's Field Extraction Schema, read the PDF, extract all schema fields, and return structured output matching insurance.policies[] format."
- Stateless by design, read-only tools only
- Anti-hallucination: "If a field cannot be found in the document, set it to null and flag as [NOT FOUND]. NEVER guess or infer field values."
- GDPR note: "Policy documents may contain personal data. The agent reads but NEVER persists document content. Only the extracted structured fields are returned to the orchestrator."

**execution_context section:**
  ${CLAUDE_SKILL_DIR}/references/germany/${insurance_type}.md
  ${CLAUDE_SKILL_DIR}/references/disclaimer.md

**process section (4 phases):**

Phase 1: Read Input
- Read `doc_path` and `insurance_type` from Task prompt (both required)
- Validate insurance_type against known types
- If insurance_type is "health": map to reference doc path `health-insurance.md`
- Load the reference doc and extract: Field Extraction Schema table, Keyword Map

Phase 2: Read Document
- Read the file at `doc_path` using the Read tool (handles PDF natively)
- If Read fails: output error "Could not read document at {doc_path}. Ensure the file exists and is a text-layer PDF."

Phase 3: Extract Fields
- For each field in the Field Extraction Schema:
  - Search for the German label (from "German Label in Policy" column) in the document text
  - Extract the value
  - Convert to the expected type (number, string, ISO date, string[])
  - Use the Keyword Map for terminology translation where needed
- For coverage_components: extract line items from coverage tables or Leistungsumfang sections
- For exclusions: extract from Ausschluesse sections
- Set `id` to `{insurance_type}-{provider_slug}-{year}` where provider_slug is lowercase provider name with spaces replaced by hyphens, and year is extracted from start_date
- Set `type` to the `insurance_type` parameter value

Phase 4: Format Output
- Return extracted data in `<doc_reader_result>` tags
- Structure must match insurance.policies[] schema exactly (one JSON object)
- Include all fields: id, type, provider, premium_monthly, premium_annual, coverage_amount, start_date, renewal_date, kuendigungsfrist_months, sonderkundigungsrecht, doc_path, coverage_components, notes, last_updated
- Flag any [NOT FOUND] fields
- Include extraction confidence level
- Include extraction notes for any converted or missing fields

**error_handling section:**
- Scanned/image PDFs: "This document appears to be a scanned image without a text layer. The document reader cannot extract text from image-only PDFs. Please provide a text-layer PDF or enter policy details manually via /finyx:profile."
- Unrecognized insurance_type: "Unknown insurance type '{type}'. Valid types: health, haftpflicht, hausrat, kfz, rechtsschutz, zahnzusatz, risikoleben, reise, fahrrad, kfz-schutzbrief, mietkaution."
