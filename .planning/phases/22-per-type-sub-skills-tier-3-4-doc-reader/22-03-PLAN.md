  <action>
Create `skills/insurance/sub-skills/doc-reader.md`. This is an ORCHESTRATOR sub-skill — it does NOT extract PDF content itself. It coordinates folder scanning, agent spawning, batch review, and profile writing.

- **Heading:** `# Insurance Document Reader Sub-skill`
- **Objective:** Parse insurance policy PDFs from a configured folder, extract structured data via the doc-reader agent, present batch results for confirmation, and save confirmed extractions to insurance.policies[]. Uses Write tool — requires user confirmation before every write.

**Phase 0: Document Location Resolution**

Read `.finyx/profile.json`. Check `documents.locations.insurance`.

If `documents.locations.insurance` is not null and not empty string:
  - Set `docs_folder` to the value
  - Emit: "Using configured insurance document folder: {docs_folder}"

If `documents.locations.insurance` is null OR the `documents` section is absent:
  - Use AskUserQuestion (text input): "Where are your insurance policy documents stored? Enter the folder path (e.g., /Users/you/Documents/insurance or ~/Dropbox/Versicherungen). This path will be saved to your profile for future use."
  - Set `docs_folder` to user input
  - Use AskUserQuestion (singleSelect): "Save '{docs_folder}' as your insurance document folder in your profile?" → "Yes, save for future use" / "No, use for this session only"
  - If "Yes": Read `.finyx/profile.json` in full, update `documents.locations.insurance = docs_folder` (create `documents.locations` section if absent), write the FULL JSON back using Write tool. Confirm: "Saved document folder to your profile."

**Phase 1: Folder Scan**

Run via Bash tool:
```bash
ls "{docs_folder}"/*.pdf 2>/dev/null
```

If zero PDFs found: emit error and STOP:
```
No PDF files found in {docs_folder}.
Ensure the folder contains .pdf files and the path is correct.
```

Present found files as a numbered list. Use AskUserQuestion (multiSelect): "Which PDF files would you like to process?" with all found PDFs listed plus a "Process all" option.

**Phase 2: Disclaimer**

Emit banner:
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 FINYX ► INSURANCE ► DOCUMENT READER
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

Emit disclaimer.md content (loaded by router). Then append:
```
Extracted data will be reviewed before saving.
Nothing is written to your profile without your explicit confirmation.
```

**Phase 3: Type Auto-detection + Agent Spawn (per PDF)**

For each selected PDF:

1. **Filename-based type detection:** Check if filename contains any known slug (case-insensitive substring match against: haftpflicht, hausrat, kfz, rechtsschutz, zahnzusatz, risikoleben, reise, fahrrad, schutzbrief, mietkaution, health, kranken).
   - Map matches: "schutzbrief" → "kfz-schutzbrief", "kranken" or "health" → "health", all others → direct slug match
   - If multiple slugs match: use the longest match

2. **If filename match fails:** Use AskUserQuestion (singleSelect) for THIS file: "What type of insurance document is '{filename}'?" with all 11 types listed plus "Skip this file".

3. **Spawn agent:** Use Task tool to spawn `finyx-insurance-doc-reader-agent` with:
   ```
   You are the finyx-insurance-doc-reader-agent. Extract structured policy data from the provided document.

   doc_path: {full_path_to_pdf}
   insurance_type: {detected_or_selected_type}

   Return your output wrapped in <doc_reader_result> tags.
   ```

4. **Collect result:** Parse `<doc_reader_result>` output. If agent returns an error (scanned image, unreadable file): record as failed extraction with error message.

**Phase 4: Batch Review**

Present all extraction results in a summary table:

```
## Extraction Results

| # | File | Type | Provider | Coverage | Premium/mo | Confidence | 
|---|------|------|----------|----------|------------|------------|
| 1 | huk-haftpflicht.pdf | haftpflicht | HUK-COBURG | EUR 5,000,000 | EUR 8.50 | HIGH |
| 2 | reise-2024.pdf | reise | ERGO | Service-based | EUR 12.00 | MEDIUM [!] |
| 3 | scan-old.pdf | — | FAILED | — | — | — |
```

`[!]` = one or more fields are LOW CONFIDENCE or NOT FOUND — review before saving.

For failed extractions: show error reason in a separate "Failed Extractions" section.

For LOW confidence extractions: group in a "Needs Review" section with warning: "[!] Low confidence extraction — verify all fields before relying on this data."

Use AskUserQuestion (multiSelect): "Which extractions would you like to save to your profile?" — list only successful extractions.

**Phase 5: Profile Write**

For each confirmed extraction:

1. Read `.finyx/profile.json` in full (re-read before EACH write to avoid stale state)
2. Check `insurance.policies[]` for existing entry with same `type` AND `provider` (case-insensitive match)
3. If duplicate found: Use AskUserQuestion (singleSelect): "An existing {type} policy from {provider} was found in your profile. Update it with the new extraction?" → Yes / No (skip)
4. If "Yes" or no duplicate: construct the policy JSON object from `<doc_reader_result>` extracted fields
5. Append to (or replace in) `insurance.policies[]`
6. Write the FULL profile.json back using Write tool

After all writes: emit confirmation count: "Saved {N} policies to your profile."

**Phase 6: Summary**

List all saved policies with key fields (type, provider, coverage, premium).
Note any skipped or failed extractions.
Suggest: "Run `/finyx:insurance portfolio` to see your updated coverage overview."

- **Error handling:**
  - No profile: same pattern as other sub-skills
  - Empty folder: "No PDF files found in {docs_folder}."
  - All files fail (scanned images): "No text-layer PDFs found in {docs_folder}. The document reader requires text-based PDFs. Scanned/image PDFs cannot be processed. Enter policy details manually via `/finyx:insurance portfolio`."
  - Partial failure: show successful + failed in separate groups

- **Notes:**
  - This sub-skill WRITES to profile.json (unlike type sub-skills which are read-only)
  - Every Write call MUST be preceded by AskUserQuestion confirmation
  - Full JSON read-update-write cycle to avoid partial corruption
  - Doc reader agent already supports all 11 insurance types — no agent changes needed
  - Scanned/image PDF detection is handled by the agent — this sub-skill just surfaces the error

- **NO YAML frontmatter, NO execution_context, NO `---` separators.**
