**`<process>` section with these phases:**

**Phase 1: Read Profile Data**
- Read `.finyx/profile.json`
- Extract: `identity.residence_country`, `identity.cross_border`, `identity.family_status`
- Extract: `investor.income.total` (gross), `investor.marginalRate`, `investor.liquidAssets`, `investor.monthlyCommitments`
- Extract: `countries.germany.gross_income`, `countries.germany.marginal_rate` (if DE)
- Extract: `countries.brazil.gross_income` (if BR and cross_border == true)
- Extract: `goals.primary_goals[]`
- If `.finyx/config.json` exists, check for `allocation_mapping` key (D-07 subsequent runs)

**Phase 2: Check Allocation Mapping (D-07)**
- If `allocation_mapping` exists in `.finyx/config.json`: use stored mapping, skip to Phase 3
- If no stored mapping (first run):
  1. Read `investor.monthlyCommitments` and `goals.primary_goals` as signal
  2. Infer category mapping using Finyx taxonomy: Needs (rent, utilities, groceries, insurance, transport), Wants (dining, subscriptions, entertainment, travel), Savings (savings accounts, Tagesgeld), Investments (broker accounts, ETFs, pension contributions), Debt (mortgage, personal loans)
  3. Present inferred mapping to user and ask for confirmation
  4. Include confirmed mapping in output under `<allocation_mapping_confirmed>` sub-tag for orchestrator to persist to `.finyx/config.json`
- Note in prompt: "You cannot write files. Return the confirmed mapping in your output — the orchestrating command will persist it."

**Phase 3: Compute Net-After-Mandatory Income**
- Use benchmarks.md Section 1 formulas — do NOT hardcode rates
- For Germany: Gross minus employee social contributions (~19.6%) minus income tax (use `marginal_rate` from profile). Reference benchmarks.md for contribution breakdown.
- For Brazil (if cross_border): Gross minus INSS (use bracket table from benchmarks.md) minus IRRF withholding. Reference benchmarks.md for BR formula.
- Output the net-after-mandatory figure per country

**Phase 4: Compute Allocation Breakdown**
- Using the allocation mapping (stored or confirmed), categorize `investor.monthlyCommitments` into Needs/Wants/Savings/Investments/Debt
- Note: profile only has a single `monthlyCommitments` total — agent must explain this is the total commitments figure and break down what is known from profile fields (e.g., broker contributions go to Investments, liquid assets imply Savings level)
- Compute each category as % of net-after-mandatory income
- Compare against benchmarks.md Section 2 adjusted 50/30/20 ranges per country:
  - DE: Needs 40-55%, Wants 15-25%, Savings+Investments 20-30%, Debt 0-15%
- Apply traffic-light thresholds from scoring-rules.md ALLOC-01

**Phase 5: Emergency Fund Check**
- From scoring-rules.md ALLOC-02: gap = (6 x monthly_expenses) - investor.liquidAssets
- monthly_expenses = investor.monthlyCommitments (proxy)
- If gap > 0: flag with traffic light per scoring-rules.md thresholds
- If gap <= 0: GREEN, state months covered = liquidAssets / monthlyCommitments
- For cross-border users: use 6-month target (not 3-month) per benchmarks.md Section 3
- Output in traffic-light format:
  ```
  [COLOR] Emergency Fund (COUNTRY): X.X months covered — target 6 months
    Gap: EUR X,XXX to reach 6-month target
    How to close: Build EUR X,XXX liquid savings in Tagesgeld
  ```

**Phase 6: Format Output**
- Wrap entire result in `<allocation_result>` tags (per D-01)
- Include income breakdown table with columns: Category | Monthly | % of Net-After-Mandatory | Benchmark | Status
- Include emergency fund traffic-light block
- If first run (D-07), include `<allocation_mapping_confirmed>` sub-tag with the confirmed mapping in YAML format
- Append confidence level: HIGH if all fields present, MEDIUM if monthlyCommitments is only proxy, LOW if key fields missing
- Append disclaimer reference
